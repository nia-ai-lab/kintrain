# KinTrain AIメニュー生成 仕様書

最終更新日: 2026-03-06
対象: 設計レビュー用
ステータス: 実装前

## 1. 目的

- `AIメニュー生成` 画面を、単純なモック提案画面から、AIとの対話でトレーニングメニューセットを作る機能へ拡張する。
- AIが提案した内容を、既存のメニューセットや既存メニューを変更せず、新規メニューセットとして登録できるようにする。
- AIによる提案内容の作成と登録を、AgentCore Runtime + Gateway(MCP) で実現する。

## 2. 画面

- 対象画面: `/training-menu/ai-generate`
- 役割:
- AIにメニュー作成条件を入力する
- AIから提案をチャット形式で受ける
- 追加の指示で提案内容をブラッシュアップする
- AIが確定提案したメニューセットを新規登録する

## 3. 入力要件

### 3.1 初回入力フォーム

- 方針:
- `マシンのみ`
- `マシン + フリーウェイト`
- `フリーウェイトのみ`
- 目標:
- `筋肥大`
- `減量`
- `維持`
- 週間頻度:
- 数値 `1..7`
- ジム施設入力:
- `ジム名称` または `設備説明URL`
- 自由入力欄:
- AIへの個別要求を記述する任意テキスト

### 3.2 入力制約

- 方針、目標、週間頻度は必須
- ジム施設入力は必須
- 自由入力欄は任意
- 週間頻度は整数のみ
- ジム施設入力は1つのテキスト欄で受け付ける
- URL形式であればURLとして扱い、それ以外は名称として扱う

## 4. AIへの依頼方式

### 4.1 UI側で付与する固定指示

- UIは、ユーザ入力に加えて、`これはトレーニングメニュー作成依頼である` ことを説明する固定指示を自動付与して Runtime に送る
- Runtime は通常のAIチャットと同じ実装を利用し、メニュー生成用の専用モード切替は持たない
- メニュー生成かどうかの制御は、UIが付与する固定指示と会話内容で行う
- UIから送るのは以下:
- 初回条件
- ジム施設入力
- 自由要求
- 既存の会話履歴
- 固定メニュー生成指示
- `userProfile`
- `aiCharacterProfile`

### 4.2 AIとの会話方式

- 表示形式は一覧プレビューではなくチャット形式とする
- 初回入力送信後、AIが説明と提案メニュー案を返す
- ユーザは追加指示を送って内容をブラッシュアップできる
- 何度か往復した後、ユーザが登録を指示したタイミングで、AIがメニューセットを登録する

## 5. チャット表示要件

- 画面上部に初回条件入力セクションを持つ
- 条件送信後は、AIチャットUIを主表示とする
- AIメッセージでは以下を自然文として表示できること
- 提案方針の説明
- 採用した前提
- 推奨メニュー一覧
- 種目の理由
- 不足設備や追加質問
- ユーザが追加指示を送れる通常のチャット入力欄を持つ
- チャットセッションは通常AIチャットとは分離する
- 条件を変更しない限り、同一セッションでAIと提案内容を詰めていく
- 条件を変更した場合は、新規セッションを開始する

## 6. 登録要件

### 6.1 登録単位

- AI提案を登録する場合は、必ず `新規トレーニングメニューセット` として登録する
- そのメニューセット内に含まれる各 `TrainingMenuItem` も、必ず新規作成する

### 6.2 既存データ非破壊

- 既存の `TrainingMenuSet` を更新しない
- 既存の `TrainingMenuItem` を更新しない
- 既存のメニューセットへ種目追加しない
- 既存メニューの `displayOrder` や `itemIds` を変更しない

### 6.3 AI生成フラグ

- AIが生成・登録した `TrainingMenuItem` は、すべて `isAiGenerated = true` とする
- 人手で作成した既存メニューは変更しない

### 6.4 メニューセット名

- AIが提案したメニューセット名をそのまま使う
- ただし空文字は不可
- 名前重複時の扱いは実装設計で定義する

## 7. MCP利用要件

- AIによる登録処理は MCP 経由で実行する
- Runtime から Gateway を呼び出し、Lambda でメニューセットとメニュー登録を行う
- UIから Core API を直接叩いて登録する方式は採用しない

## 8. 想定MCPツール

- `search_gym_facility`
- 入力: `gymInput`
- 役割: ジム名称またはURLから設備情報取得方針を決める
- `create_training_menu_set_from_ai`
- 入力: `setName`, `items[]`, `isDefault?`
- 役割: 新規メニューセットと新規種目を一括登録する
- `preview_training_menu_set_from_ai`
- 入力: `draft proposal`
- 役割: AI内部で構造化データを明示するための論理上の中間モデル

注記:
- 実際に永続化するMCPツールは `create_training_menu_set_from_ai` が中心となる
- `preview_*` はUIに返す構造化出力設計として必要な場合のみ採用する

## 9. AI出力要件

- AIはユーザ向け説明テキストに加えて、登録可能なメニューセット構造を内部的に保持できること
- ユーザが `この内容で登録して` など明示指示した場合のみ、登録ツールを呼び出す
- AIが勝手に登録を実行してはならない

## 10. 失敗時要件

- ジム設備情報が取得できない場合は、その旨を説明し、設備不明前提で仮案を出すか、追加情報を要求する
- MCP登録失敗時は、どの工程で失敗したかをユーザへ説明する
- 一部だけ登録される中途半端状態は避ける
- 登録処理は論理的に一括成功/失敗になるよう設計する

## 11. 現時点の前提

- 初回の登録対象は `TrainingMenuSet` と `TrainingMenuItem` のみ
- AIメニュー生成画面から `Goal` や `DailyRecord` は更新しない
- 登録後に自動でその新規メニューセットをアクティブ表示対象にするかは別途決定する

## 12. 非対象

- 既存メニューの差分更新
- 既存メニューセットへのマージ
- AI提案メニューの部分採用UI
- UI側だけでの仮登録
